# GenAI-Assisted Development вЂ” Repo Template Concept

Concept for a repository template to be rolled out across multiple repos, providing a consistent structure for
developing software with GenAI coding agents (Claude Code etc.).

## 1. Overview

The template has two categories of content with different lifecycles:

| Category | Changes | Owned by | Mechanism |
|---|---|---|---|
| Skills / commands / agents / AGENTS.md core | Frequently, centrally | Architecture community | Shared toolkit repo, synced via submodule |
| docs / sessions / tasks skeleton | Rarely, structurally | Per-repo, once rolled out | Copier template, scaffolded once |

## 3. Decisions made

### 3.1 Harness extensions (`.claude/`)
- Skills, commands, and agents are split into **shared** (org-wide, via
  `.claude/shared/`) and **local** (repo-specific).
- Shared content lives in a **central toolkit repo** (`shs-ct-agent-toolkit`)
  and is consumed via **git submodule**, pinned to a version tag
  (`vX.Y.Z`) вЂ” updates are a deliberate one-line pin bump per repo, not
  a forced sync.
- **Local wins** on a name collision between shared and local
  skills/commands/agents; collisions should be flagged (e.g. CI check),
  not silently shadowed.
- Naming convention: shared items namespaced (e.g. `sct-<name>`), local
  items unprefixed, so the origin is visible even outside the folder
  structure.
- **Promotion path**: a repo-local skill/command/agent that proves
  broadly useful is promoted via a **direct PR against
  `shs-ct-agent-toolkit`** вЂ” same review bar as any other shared code,
  no separate staging area.

### 3.2 AGENTS.md / CLAUDE.md
- `AGENTS.md` is the tool-agnostic shared source of truth (conventions,
  build/test commands, architecture pointers).
- `CLAUDE.md` stays thin, referencing `AGENTS.md` plus Claude-Code-specific
  notes вЂ” avoids duplication drift.

### 3.3 Documentation (`docs/`)
- Organized using the **Diataxis** framework: `tutorials/`, `how-to/`,
  `reference/`, `explanation/`.
- Architecture decisions recorded as **ADRs in `docs/decisions/`**, using
  the **MADR** (Markdown Architectural Decision Records) template.

### 3.4 Session tracking (`sessions/`)
- **One file per logical unit of work** (not one per agent conversation)
  вЂ” reopened/appended to while in progress, rather than starting a new
  dated file per conversation.
- Filename dated by the **start** of the unit of work; stays stable for
  the file's lifetime so links to it remain valid.
- Template structure:
  - Frontmatter: `started`, `updated`, `status` (`in-progress` |
    `completed` | `blocked`), `task`, `adr`, `agent`.
  - `## Goal` вЂ” written once, stable.
  - `## Context` вЂ” links only, no restating.
  - `## Current state` (while in-progress/blocked) вЂ” **rewritten in
    place** each session: `Open questions / risks`, `Next steps`.
  - `## Outcome` вЂ” **replaces** `Current state` once `status: completed`;
    final, frozen record of what shipped/was decided.
  - `## Log` вЂ” append-only, one dated entry per work session (what
    happened, decisions made, files touched).
- `INDEX.md` is a flat table (date, session, status, summary), sorted by
  `updated` (not `started`), so recently-touched in-progress work
  surfaces first.
- **Cold-start read protocol** is wired into `AGENTS.md`: before starting
  work, read `sessions/INDEX.md`; if the latest relevant entry is
  `in-progress`/`blocked`, read that file in full first.
- Generated by a shared command (`.claude/shared/commands/session-log`),
  drafted by the agent from its own context, human-reviewed before
  commit.

### 3.5 Task tracking (`tasks/`)
- **Directory-as-status** (Backlog.md-style): `backlog/` в†’ `in-progress/`
  в†’ `done/`. Moving a task is a `git mv`, giving a free audit trail.
- Task file frontmatter: `id` (slug), `status`, `priority`, `created`,
  `owner`. Body: `Description`, `Acceptance criteria`, `Related` (links
  to ADR and session log via the shared slug).
- The same **slug** is used across task, session log, and ADR, so all
  three can be found via a single grep.
- `BOARD.md` is a **generated** summary view (in-progress / backlog /
  done), never hand-maintained вЂ” produced by a shared command
  (`.claude/shared/commands/task-board`).
- **Deferred**: whether/how `tasks/` links to GitHub Issues or another
  work-item system of record. Considered purely additive (e.g. an
  `external_id:`/`github_issue:` frontmatter field) вЂ” not blocking
  initial rollout, but slugs should be chosen with a future link in mind.

## 4. Implementation checklist

### Toolkit repo (`shs-ct-agent-toolkit`)
- [ ] Create `shs-ct-agent-toolkit` repository
- [ ] Set up `skills/`, `commands/`, `agents/` directories
- [ ] Write initial `AGENTS.md` shared fragment
- [ ] Define versioning/tagging scheme for the toolkit repo
- [ ] Define contribution/review process for promoted skills

### Copier template
- [ ] Set up Copier template repository/structure
- [ ] Template `AGENTS.md` header (repo name, team, links) + shared fragment include
- [ ] Template thin `CLAUDE.md`
- [ ] Scaffold `.claude/` (submodule wiring for `shared/`, empty local dirs)
- [ ] Scaffold `docs/` Diataxis quadrants + `decisions/` with MADR template file
- [ ] Scaffold `sessions/` with `INDEX.md` template
- [ ] Scaffold `tasks/` with `backlog/`, `in-progress/`, `done/`, empty `BOARD.md`

### Shared commands
- [ ] Build `session-log` command (draft session entries from agent context)
- [ ] Build `task-board` command (generate `BOARD.md` from `tasks/`)
- [ ] Decide collision-check mechanism for shared vs. local skills/commands/agents

### Process / rollout
- [ ] Wire cold-start read protocol into `AGENTS.md` (read `sessions/INDEX.md` first)
- [ ] Pilot the template on one repo end-to-end
- [ ] Document submodule pin-bump process for consuming repos
- [ ] Roll out to remaining SHS-CT repos

## 5. Open points (not yet discussed)

- Toolkit repo versioning/release process in detail (branching, tagging
  cadence, changelog).
- How consuming repos pin and bump the `.claude/shared/` submodule in
  practice (manual PR vs. automated bump bot, review requirements).
- Whether/how to link `tasks/` to GitHub Issues or another work-item
  system of record (explicitly deferred).
- ADR numbering/lifecycle details beyond "MADR format" (e.g. superseding
  ADRs, status field conventions).
- Governance: who can edit the shared toolkit vs. repo-local content;
  how the architecture community reviews promoted skills.
- Onboarding: how a new repo (or new team member) is introduced to the
  template and its conventions.
- Tooling for enforcing structure (e.g. CI checks for collisions, stale
  `BOARD.md`/`INDEX.md`, missing frontmatter).
- Relationship to IEC 62304 / EU MDR regulated-process documentation вЂ”
  whether any of this structure needs to interface with the formal
  quality system, or stays purely as an agent-assistance layer.
